## ✨ What makes it different?

In one sentence:

**VibeSkills is not asking you to memorize 340+ skill names. It is trying to make AI work more stable, more understandable, and easier to trust.**

Think of it as an AI workbench that pauses to understand the job before it starts doing the job.

| Real problem | What VibeSkills does |
|:---|:---|
| AI starts too fast and drifts off target | Clarify first, then plan, then execute |
| Too many skills, no idea which one to use | Let the router pick the primary skill |
| Different workflows step on each other | Keep one main flow per task, use other skills as support |
| Every new session feels like starting over | Reconnect context through workspace memory |

The core facts are simple:

- `340+` callable skills
- `19+` high-value upstream projects absorbed into the system
- `129` governance rules used as guardrails


## 👥 Who is it for?

This repo fits best if you are:

- using AI for real project work, not just one-off questions
- trying to make AI work traceable, reviewable, and handoff-friendly
- building a stable AI workflow for yourself or a small team
- tired of collecting skills that never work together in practice

If you only want one tiny utility script, this may be too much. If you want a fuller and more reliable AI workflow, this is what the repo is built for.


## 🔀 Intelligent Routing: How 340+ Skills Collaborate Without Conflict

Plain-English version:

After you describe the task, the system decides whether this is mainly planning, implementation, debugging, review, or research work. It then chooses one best-fit primary skill to lead the flow. Other skills can still help, but they come in as supporting steps instead of all activating at once.

You can think of the flow like this:

1. You say what you want done.
2. The router identifies the main task type.
3. If the task is complex, the system clarifies scope first.
4. It chooses the execution intensity with `M / L / XL`.
5. It verifies the result before claiming completion.

This design is mainly there to guarantee four things:

- **You do not need to memorize the skill catalog**.
- **You do not get multiple competing main workflows**.
- **Complex work gets scoped before code changes start**.
- **Delivery requires verification, not just confidence**.

### M / L / XL Execution Levels

These are not different products. They are just different execution intensities:

<div align="center">

| Level | Use Case | Characteristics |
|:---:|:---|:---|
| **M** | Small, clear tasks | Single agent, fast, token-efficient |
| **L** | Medium complexity work that needs planning and review | Moves through the plan step by step |
| **XL** | Larger, longer, multi-agent work | Runs in waves and only parallelizes clearly independent units |

</div>

If you want to state a preference, you can say:

```text
Please handle this with the XL workflow /vibe
```

<details>
<summary><b>🔍 Want the more technical routing notes? Click to expand</b></summary>

<br/>

- There is one authoritative router. Two runtimes do not take over the same task.
- A task normally gets one primary skill. Other skills stay scoped as helper phases.
- The only lightweight public overrides are `--l` and `--xl`.
- `vibe`, `vibe-want`, `vibe-how`, and `vibe-do` still resolve to the same runtime underneath.

</details>


## 🧠 Memory System: AI That Truly Remembers

The goal of this part is simple:

**do not make AI act like it lost its memory every time you open a new session.**

In practice, it addresses problems like these:

<div align="center">

| Problem you run into | What the system does |
|:---|:---|
| You have to repeat project background in a new session | Store project decisions and conventions in project-level memory |
| Long tasks lose early context halfway through | Pull relevant context back with in-session semantic retrieval |
| Useful lessons disappear after the task ends | Save important conclusions into longer-term memory |
| A new agent has to take over | Fold context into a structured handoff summary |

</div>

You only need to remember four ideas:

- **shared memory inside the same workspace**
- **strict isolation across different workspaces**
- **retrieve only relevant memory**
- **fail loudly instead of silently downgrading**

If you want the layer model, this is the short version:

| Tier | Component | Scope | Main purpose |
|:---:|:---:|:---:|:---|
| **L1 Session** | `state_store` | Current session | Progress, intermediate results, temporary state |
| **L2 Project** | `Serena` | Current project | Architecture decisions, conventions, project rules |
| **L3 Short-term Semantic** | `ruflo` | Within the session | Fast retrieval of relevant context during long tasks |
| **L4 Long-term Graph** | `Cognee` | Across sessions | Entity links, relationships, long-horizon knowledge |

### What the workspace-shared memory upgrade changes in practice

This release makes memory continuity workspace-based by default. In practical terms:

- **Same workspace, different sessions or agents can share memory**.
- **Different workspaces stay isolated even if they use the same backend root**.
- **Search is based on business-relevant terms, not generic scaffold words like `$vibe` or `plan`**.
- **Early stages only receive a small memory capsule, not a full history dump**.
- **If the workspace broker is unavailable, the runtime fails explicitly instead of silently falling back**.

You can picture the flow in four steps:

1. The runtime stores small structured records such as decisions, handoff cards, and relations.
2. Follow-up work searches only inside the current workspace.
3. The system injects a small number of bounded memory capsules into the next stage.
4. Later stages can reveal more, but the entire memory store is never dumped into the prompt.

The technical contract is in [workspace memory plane design](./docs/design/workspace-memory-plane.md), and the benchmark-oriented validation is in [quantitative Codex memory simulation](./tests/runtime_neutral/test_codex_memory_user_simulation.py).

<details>
<summary><b>🔧 Need the advanced install notes? Click to expand</b></summary>

<br/>

**Manual follow-up uses real host paths**

When a follow-up step stays manual, the install docs now point to the real host-managed path instead of hand-waving. Examples:

- Codex: `~/.codex/settings.json`
- Claude Code: `~/.claude/settings.json`
- Cursor: `~/.cursor/settings.json`
- OpenCode: `~/.config/opencode/opencode.json`
- Windsurf / OpenClaw sidecar state: `<target-root>/.vibeskills/host-settings.json`

**What the README is claiming after install**

The current topology has probe coverage against the installed runtime, not only repo-local scripts.

| Host | Installed public entry | Installed-runtime probes covered |
|:---|:---|:---|
| `codex` | installed `skills/vibe` entry | planning, debug, governed execution, memory continuity |
| `claude-code` | managed closure + installed `skills/vibe` payload | planning, debug, governed execution, memory continuity |
| `openclaw` | runtime-core adapter + installed `skills/vibe` payload | planning, debug, governed execution, memory continuity |
| `opencode` | preview-guidance adapter + installed `skills/vibe` payload | planning, debug, governed execution, memory continuity |

Those probes are intended to show that the installed runtime still owns routing, emits governance artifacts, writes cleanup receipts, and preserves memory continuity. They are not claiming that every host-native invocation surface was directly exercised in the same probe.

**Public topology in one glance**

- public runtime entry: `<target-root>/skills/vibe`
- internal bundled corpus: `<target-root>/skills/vibe/bundled/skills/*`
- compatibility projections: only when a host explicitly needs them

The `.vibeskills` layout is intentionally split:

- host-sidecar: `<target-root>/.vibeskills/host-settings.json`, `host-closure.json`, `install-ledger.json`, `bin/*`
- workspace-sidecar: `<workspace-root>/.vibeskills/project.json`, `.vibeskills/docs/requirements/*`, `.vibeskills/docs/plans/*`, `.vibeskills/outputs/runtime/vibe-sessions/*`

**Uninstall / cleanup**

`uninstall.ps1 -HostId <host>` and `uninstall.sh --host <host>` are the matching uninstall paths. By default they perform ledger-first, owned-only cleanup. The deeper governance notes are in [`docs/uninstall-governance.md`](https://github.com/foryourhealth111-pixel/Vibe-Skills/blob/main/docs/uninstall-governance.md).

**Custom skills**

See the [custom workflow & skill onboarding guide](https://github.com/foryourhealth111-pixel/Vibe-Skills/blob/main/docs/install/custom-workflow-onboarding.en.md).

</details>
